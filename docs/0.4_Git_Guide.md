# Git Guide

## Ziel
Das Projekt soll möglichst simpel mit Git geführt werden: nur ein Branch, keine Feature-Branches, nur Commit und am Ende Tagging.

## Commit-Richtlinien
- Commit-Messages beginnen immer damit, worum es geht: entweder "Doku: …" für Dokumentationsänderungen oder "Code: …" für technische Anpassungen.
- Jeder Commit fasst eine abgeschlossene Änderung zusammen (kein halbfertiger Zustand).
- Bei Bedarf inhalt kurz beschreiben, z. B. `Doku: Kapitel 3.2 erweitern` oder `Code: Berechnung der Heizlast angepasst`.

## Tagging
- Tags werden nur dann gesetzt, wenn wir einen Meilenstein oder eine Version fertig haben.
- Der erste Tag ist `0.1.0` (entspricht dem aktuellen Stand); neue Tags vergebt ihr, sobald eine neue Version veröffentlicht wird.

## Workflow (lokal)
1. `git status` prüfen, um weitere Änderungen einzusehen.
2. `git add <Datei>` (oder `git add .` bei mehreren Dateien).
3. `git commit -m "Doku: …"` oder `git commit -m "Code: …"` entsprechend dem Inhalt.
4. Falls alles bereit ist, nach GitHub pushen (`git push origin main` oder wie der zentrale Branch heißt).
5. Zum Abschluss ein Tag setzen, z. B. `git tag 0.1.0` und anschließend `git push origin 0.1.0`

## Hinweise
- Keine Branches oder Merge-Strategien nötig, solange wir alle auf `main` arbeiten.
- Dokumentation und Codeänderungen sollten sauber voneinander getrennt beschrieben werden, damit man in der Historie nachvollziehen kann, was geändert wurde.
